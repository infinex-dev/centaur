Active Validator + History Guards

This pass ports the useful Nigel pattern into comms-factory without replacing the existing generator/validator path. The new path is opt-in and keeps the ReleaseCard as the hard fact boundary.

implemented behind flag tests passing

Enable in harness: set HARNESS_ACTIVE_VALIDATOR=1.

Optional caps: COMMS_ACTIVE_VALIDATOR_MAX_TOOL_TURNS, COMMS_ACTIVE_VALIDATOR_MAX_TOKENS, COMMS_ACTIVE_VALIDATOR_MODEL.

Pipeline Shape

brief
  -> fact grounder
  -> operator-approved ReleaseCard
  -> generator
  -> deterministic validator
  -> active research validator
  -> history guards
  -> orchestrator pick
  -> operator approve/edit/reject
  -> ship ledger / copy history

What Changed

  • Shared research tools
    • src/research-tools.ts now owns the reusable research tool schemas and executor.
    • src/fact-grounder-llm.ts now imports those shared tools instead of owning a separate copy.
    • Shared tools include platform grep/read, partner registry, Infinex page fetch, rendered page fetch, JSON API fetch, ProjectJin web search, and recent X search.
  • Active validator
    • src/validator-active.ts adds Nigel-style tool-use validation.
    • It runs deterministic validation first. Regex/fact-contract failures short-circuit before any model/tool spend.
    • The active validator can research contradiction/staleness, but it cannot expand allowed claims.
    • If research finds a true fact missing from card.deployed_facts, it fails with card_missing_fact.
  • History guards
    • src/history-guards.ts adds deterministic copy-history checks.
    • X gets stricter defaults for repeated openers, phrase budgets, primary-tempo cadence, rare tempo reuse, announcement-frame saturation, and optional protagonist repetition.
    • General copy can use the same guard with softer budgets by channel.
  • Active orchestration
    • src/orchestrator.ts now exports orchestrateActive() and orchestrateActiveWithRetries().
    • Retry feedback now includes active-validator failures and history-guard failures.
    • The existing orchestrate(), orchestrateLLM(), and orchestrateWithRetries() paths remain in place.
  • Harness persistence and UI
    • harness/lib/schema.sql adds candidate_audits.
    • harness/app/actions/generate.ts uses the active path when HARNESS_ACTIVE_VALIDATOR=1.
    • harness/lib/queries.ts joins the latest active audit into candidate records.
    • harness/components/CandidateCard.tsx shows active-validator and history-guard pass/fail state, turn count, tool count, and failure reasons.

State Model

Candidate row

Still stores the original deterministic validator output: text, declared beats, beat audit, validation pass/fail, failures, rationale, source.

Candidate audit row

Stores optional active audit and history guard JSON keyed by candidate id. This avoids mutating the old candidate table contract.

History source

The harness builds recent-copy context from prior final_picks on the same channel, excluding the current card.

Caveats

  • restart needed If the Next harness server was already running, restart it so it loads the new server action code and schema helpers.
  • evaluated candidates only The active audit is stored for candidates the active orchestrator actually evaluated while searching for a pick. Later unevaluated candidates from the same batch may have no active audit row.
  • fact boundary preserved The active validator does not edit or enrich ReleaseCards. It fails and tells the operator when the card is missing a needed fact.
  • cost guard is light This adds model token/tool-turn caps, not Nigel-style dollar accounting. That is intentional for now because comms generation is less time-sensitive and still human-gated.
