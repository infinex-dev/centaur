INFINEX // PLATFORM DOGFOOD CAPTURE

What The May Dogfood Found, And What To Do About It

Forty minutes of recorded screen-walkthrough across the Infinex platform: marketing site, onboarding, wallet import, dashboard, Earn, Switch, Perps, Send, Deposit, Gas Account, and Settings. This is the working triage list for the team meeting — every issue captured, severity-tagged, evidence-anchored, and shaped so that for each row a colleague can decide fix, defer, sunset, clarify, or not a bug.

Status: triage capture No implementation approval Evidence: transcript + recording Captured 2026-05-15
Core conclusion

The product has strong primitives. The interface doesn't behave like it knows that. Old campaign residue still occupies default user surfaces, marketing copy has leaked into utility and security flows, implementation details are being shown to users instead of translated, and core Infinex primitives like the Gas Account and non-EVM deposit support are being discovered too late — or never.

Three of the findings are ship-blockers: a seed-phrase input shown in clear text on import, imported wallets that can't access HYPE, and a Send-to-self error instead of an auto-route to Move. Everything else is a stack of credibility leaks and ergonomic friction.

01 / Executive cut

Three Ship-Blockers, Twenty-Two High-Impact Items

Everything not on this short list is in the per-area tables below. If we ship only what's on this page in the next two sprints, we close the trust gap and the credibility gap simultaneously.

P0 — Ship-blockers

01

Seed phrase / private key shown in clear text on import

During wallet import the secret field is fully visible. Other wallets (Rabby, MetaMask) blank it by default and require an explicit reveal.

"It kind of does that, and it blanks it out, whereas this shows it in clear text. That's my first number one. I really don't like that."

DF-001 · type: SEC · area: Wallet Import · evidence ~ 00:07:00–00:09:00

02

Imported wallets can't route into HYPE / Hyperliquid

From an imported wallet, the platform blocks HYPE trades with "This wallet doesn't support Hype." The wallet does support it elsewhere — Infinex doesn't support it from this surface. Breaks the one-account / any-venue thesis.

"Truly one of the worst features… This is one of the worst parts about the flow."

DF-002 · type: BUG + DEAD · area: Switch / Perps · evidence ~ 00:19:00–00:21:00

03

Send-to-self errors instead of routing to Move

Pasting an owned Infinex address into Send returns: "You can't send to your own Infinex account address. Use Move instead." The app knows the destination is owned and still makes the user back out and pick a different flow.

"This is horrendous. Like, why have you let me get this far?"

DF-003 · type: DEAD + UX · area: Send · evidence ~ 00:37:00–00:38:30

04 / Tone of voice

Marketing Copy Has Leaked Into Utility, Security, And Trading Flows

Many of the P2 COPY rows above stem from one root pattern. The rule that fixes them all:

If the user is trying to complete a financial / security / trading task, do not sell them the product. Help them complete the task.

Specific words and phrases flagged in the recording, for the editorial pass:

superpowers(just describe the action)
seamless / seamlessly(omit, or describe what actually happens)

06 / Frame for the room

This Is Systemic, Not Personal

The dogfood session was blunt. The point isn't to call out individual contributors — the patterns that showed up are systemic and we can now fix them together:

  • Old campaign residue is still occupying default user surfaces (Sale, Craterun, Airdrops, Mantle, Yaprun, Points, Badges, App Brand).
  • Marketing copy is bleeding into utility, security, and trading flows, where it costs trust instead of building it.
  • Implementation details (7702 icons, venue choice on Perps, dynamic deposit addresses) are leaking out to users without translation.
  • Core Infinex primitives (gas account, non-EVM deposit support, the Move concept) are buried or discovered too late.
  • Two ship-blocker bugs (DF-001 / DF-002 / DF-003) touch security and the central "one account / any venue" thesis directly.

The product itself has strong primitives. The work is to make the interface act like it knows that.
