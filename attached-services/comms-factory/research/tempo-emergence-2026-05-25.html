What the generator is doing wrong

The violation in src/generator.ts

Stage A's BeatPlan schema puts tempo and verb as peer fields in the same JSON the LLM generates in a single pass:

generator.ts:79-85interface BeatPlan { tempo: TempoName; verb: WorkingAction; /* + thesis, through_action, obstacle, lining */ }
generator.ts:820Stage A user-prompt: "tempo": "<one of the voice's available tempi>"
generator.ts:855tempo: beat.tempo as TempoName — parsed straight through, no derivation step

The docstring at lines 66-70 claims "tempo is emergent from inner work." The prompt at line 731 tells the model "the actor never starts with a motor." But the schema forces the model to pick a tempo anyway. Intent right, implementation drifted.

Tests in generator.test.ts only verify declaration fidelity (does Stage A say what it said) and allowed-set membership (is the tempo a real one). They don't assert tempo coherence with the chosen verbs. So the violation has been silently passing CI.

The fix

Strip tempo from Stage A's output entirely. Replace with working_actions — a pair of transitive-verb identifiers per beat (Sustained-prep + Quick-release):

new BeatPlaninterface BeatPlan { working_actions: [WorkingAction, WorkingAction]; verb: string; /* + thesis, through_action, obstacle, lining */ }
new stepderiveTempoFromBeat(working_actions, voice) — deterministic motor-pair → tempo lookup

Stage B uses the derived tempo, not a Stage-A-declared one. ~80 LOC, no extra LLM round-trip.

Bonus: preparation hierarchy gets enforced naturally because Working-Actions come in Sustained-prep → Quick-release pairs:

  • Press → Punch (Sombre-ish / weighted decision)
  • Wring → Slash
  • Glide → Dab (Practical / matter-of-fact)
  • Float → Flick (Irradiant / sociable / loose)

The Quick must be preceded by its Sustained partner. If the LLM emits [Float, Punch], the validator rejects on prep-hierarchy alone — no tempo classification needed.

Validator architecture (your other question)

Layer 1 — regex

src/validator.ts (rules)

Brand-agnostic slop catchers: clichés, listicle voice, antagonism, AI-slop, em-dash density, claimed competitor palettes, off-spec drive language (e.g. time-pressure for a Stable+Flow brand). Deterministic. Fast. Free. Each rule has positive + negative tests.

Layer 2 — LLM beat audit

src/validator-llm.ts (called by validator.ts)

Independent beat classifier. Reads the candidate prose, classifies what tempo it ACTUALLY is, compares against what the generator DECLARED. If divergent → reject. This is the "verifier as gold standard" pattern. Doesn't read the declaration while scoring — so it can't be biased.

Layer 3 — agentic auditor

src/validator-active.ts

Anthropic SDK + research tools (platform-code, partner-registry, infinex-page, projectjin). Reasons about claim validity by calling tools. The deepest layer — fact + voice + drive coherence all in one.

When you walk a card through the harness, all three fire in sequence. The failures you're seeing on the generator's tempo claims are coming from Layer 2, which is exactly what should catch them.
