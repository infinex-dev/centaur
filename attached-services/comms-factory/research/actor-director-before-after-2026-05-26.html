Actor/Director Architecture: What Changed, What Still Fails

Generated 2026-05-26. This report compares the old comms-factory generator path with the new actor/director path, using the current worktree plus one live Anthropic run against a disposable harness database. Secrets are not rendered.

Architecture The system now has a real actor/director path with rollback, source-indexed memory, blind Director audits, persisted transcripts, and harness UI visibility.
Quality I am not happy calling the generator good yet. The first live web run exhausted 3 actor attempts and the Director rejected the passing candidates as acute.
Next Layer The live failure supports adding Daily Pages / Actor Notes: a persisted warm-up phase before table work and performance, invisible to the blind Director.

1. The Main Architectural Difference

Before

Per-channel generator flows

  • runGenerator looped through channels and called generateForChannel separately.
  • Each channel had its own attempt row in generator_attempts.
  • The prompt capture still spoke in Stage A / Stage B terms: inner_work_prompt_json and drafting_prompt_json.
  • Context was mostly prompt text plus voice spec. No persisted character session across channels.
  • The validator could run regex, active LLM audit, and history guard, but the generation actor and validator did not share a visible rehearsal trail.
release card
  -> generateForChannel(x)
       -> generator attempt(s)
       -> validator
  -> generateForChannel(web)
       -> generator attempt(s)
       -> validator
After

One Actor, blind Director

  • HARNESS_GENERATOR_ARCH=actor switches to runActorGenerator.
  • One Actor call handles all requested channels for an attempt.
  • The Actor receives source-indexed memory and returns table_work plus performances.
  • The Director reads candidate prose blind. It does not see actor table work.
  • Director notes are appended to the same actor transcript for retry attempts.
system: Actor Memory
user: assignment
assistant: table_work + performances
user: Director notes
assistant: revised table_work + performances

2. Persistence Difference

Before

Legacy attempt rows only

  • generator_attempts: one row per (card, channel, attempt).
  • candidate_audits: optional active validator and history guard trace.
  • candidates: text, declared beat JSON, validator result, source.
After

Additive actor/director tables

  • actor_attempts: one row per card-level Actor attempt, spanning all channels requested in that attempt.
  • director_audits: one row per blind Director read.
  • candidates stay compatible; Director fields are joined from director_audits.
  • No destructive migration. Legacy rows still render.

3. Prompt And Source Difference

Thing Before Now
Movement knowledge Mostly generated prompt sections from src/voice/infinex.ts and derived operational references. Actor and Director memory explicitly source-index the canonical Mirodan chapters, combined reference, PDF, and source-location memory.
All 24 tempi The old generator worked mainly from the Infinex palette and locked examples. ALL_TEMPO_NAMES and MIRODAN_24_TEMPI give the Actor and Director the surrounding world, not only Infinex.
Tempo selection Legacy paths could still carry declared tempo fields for beat diagnostics. The Actor output contract forbids tempo fields in beat plans. The Director classifies tempo after the fact.
Evidence Validator reasoning could include classification, but movement evidence was not source-ref structured. Director output requires motion evidence and working action evidence with source_refs. Local phrases like "single aimed sentence" are observations, not canonical claims.

6. Bug Fixed After Live Run

Before

Fact contract missed readiness claims

A sentence could contain a valid launch fact and add an unsupported readiness claim, and the old sentence-level support check would treat the whole sentence as supported. 

Fact A is live, ready to trade.

Supported? incorrectly yes, because
"Fact A is live" overlapped the deployed fact.
After

Readiness claim extractor

ready to trade, ready to use, ready to swap, and similar claims are now audited independently against deployed facts.

"ready to trade"

7. Daily Pages / Actor Notes Recommendation

Your proposed warm-up layer is the right next move. It should not replace table work. It should happen before table work and should be persisted as rehearsal material.

Current actor flow

Still too abrupt

Actor Memory
  -> Assignment
  -> Table Work
  -> Performance
  -> Director Notes
  -> Revised Table Work + Performance

This is structurally correct, but the live run shows the Actor can still perform from rules instead of lived circumstances.

Next actor flow

Warm-up before performance

Actor Memory
  -> Assignment
  -> Daily Pages / Actor Notes
       - given circumstances
       - product world
       - reader prior
       - left/right choices
       - what feels false
       - what the character refuses
  -> Table Work
  -> Performance
  -> Blind Director
  -> Director Notes
  -> More Actor Notes
  -> Revised Table Work + Performance
Design Decision Recommendation
Where to store it Add an additive actor_warmups table or columns on actor_attempts: warmup_prompt_json, warmup_response_json, given_circumstances_json, false_starts_json.
Who sees it The Actor sees it. The Director does not. The Director must stay blind to prevent self-rationalized validation.
What it should produce Not polished copy. It should produce journal pages, given-circumstances notes, rejected impulses, vocabulary that felt false, and a short distilled actor state for table work.
Why it matters It turns the Actor from "LLM applying rules" into "LLM rehearsing the character before writing." That is closer to the acting model you described.
