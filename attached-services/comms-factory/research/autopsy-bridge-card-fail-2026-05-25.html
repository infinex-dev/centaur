Autopsy — bridge card all-fail

2026-05-25 · 12 candidates / 3 attempts / 0 passes · your instinct was right: the bug is in validation, not the generator

Three-line verdict

  1. The fact-contract validator is the root cause of every failure. It does literal token matching against the full deployed_facts strings — but the model has 280 characters to carry facts that are 100-250 characters long each. The model is correctly paraphrasing; the validator can't see paraphrase as truth.
  2. The em-dash rule isn't firing even though em-dashes appear in failing candidates. The threshold is set at > 2 per 280 chars; one em-dash passes. You wanted zero tolerance. One-line fix.
  3. You can't see attempts 1 and 2 because the harness UI only shows the latest attempt's candidates. Feedback IS being passed between attempts in code (orchestrator.ts:312) — but it's not persisted, and the UI has no per-attempt accordion. That's a UI gap, not a logic gap.

Issues, ranked by impact

1. Fact-contract validator does literal matching P0 · root cause

What's happening: the validator wants the model to declare which deployed_facts it used and which it didn't (not_said). For "used" facts, it then checks the prose for at least 55% token overlap with the full deployed_fact string. For "not_said" facts, it requires an EXACT string match against the deployed_facts list.

Why this breaks: deployed_facts here are 80-250 characters long. Tweets are 280 characters. The model can't fit verbatim facts. When it paraphrases ("a permanent deposit address now lives in your account"), the token-overlap heuristic fails. When it labels facts by short name in not_said ("Bridge.xyz identity"), the literal set lookup fails. Then EVERY card fact triggers "neither deployed nor accounted for in not_said" because no entry in either array matches the full string. That's ~10 cascading failures per candidate, from one root cause.

Example from attempt 3, x channel

Candidate text:
"A permanent deposit address now lives in your account.
 Send a bank transfer to it — it arrives on-chain as USDC.
 Rails by @bridge_xyz."

deployed_fact (full 250 chars):
"Bridge.xyz fiat deposit mechanic — rail shape:
 Bridge virtual accounts are permanent, reusable fiat
 deposit addresses that convert incoming fiat into crypto
 and deliver it to a specified destination wallet address.
 Bridge handles fiat-to-crypto conversion and sends funds
 on-chain automatically."

Validator: "declared deployed fact does not appear plausibly in text"
Reality: it's a correct, compressed paraphrase. The 55%
token overlap heuristic can't fit 250 chars into 280 chars.

Fix shape (multiple options)

  • Restructure deployed_facts as fact_id + short_label + full_description. Validator's literal checks run on the SHORT label; the full description is for the LLM grounder's research, not for caption matching. Mid-size refactor, but is the clean answer.
  • Use the LLM auditor instead. The active validator (validator-active.ts) already calls Sonnet with tools to reason about claim validity semantically. If HARNESS_ACTIVE_VALIDATOR=1, that path runs. It's not on by default — candidate_audits has 0 rows for these 12 candidates. Toggle it and the literal checks become advisory rather than gating.
  • Loosen the token overlap heuristic from 55% to e.g. 25% for facts longer than 100 chars. Pragmatic patch, not the right long-term shape.

2. Em-dash rule not firing on candidates that have em-dashes P0 · easy fix

What's happening: validator.ts:118 sets EM_DASH_MAX_PER_WINDOW = 2 per 280-char window. A 285-char candidate with one em-dash has 0.5 em-dashes/window — under the threshold. So rejectAIslop passes.

Why this is wrong for Infinex: you've called out em-dashes as a hard no — they're AI-slop signature. The current rule allows up to two. Anything above zero is off-spec for the locked Infinex voice.

One-line fix

// src/validator.ts:118
- const EM_DASH_MAX_PER_WINDOW = 2;
+ const EM_DASH_MAX_PER_WINDOW = 0;

(Or stricter: reject on any em-dash unconditionally, no density math at all.)

Note: some of the deployed_facts themselves contain em-dashes ("Bridge.xyz fiat deposit mechanic — rail shape"). The grounder writes the fact strings; the rule won't fire on the facts themselves (they're not the candidate text), but the model may absorb the em-dash habit from seeing them. A second-order fix is to normalize deployed_facts to use colons or hyphens.

3. Per-attempt history invisible in harness UI P1 · UI gap

What's happening: the orchestrator runs up to 3 attempts. Between attempts, summarizeFailures(result, channels) at orchestrator.ts:312 builds a feedback string from rejected candidates' first failure and passes it to the next generateAttempt. So the generator IS getting feedback. But:

  • The feedback string is not persisted anywhere — it lives in a closure inside the orchestrator, dies at end of function.
  • The harness UI groups candidates by channel but doesn't surface the per-attempt history. You see all 12 candidates flat, no "attempt 1 → attempt 2 → attempt 3" accordion.
  • The candidate_decisions table has a retry_feedback column — but it's for operator-provided retry hints, not the auto-generated summary.

Fix shape

  • Add an attempts table or extend candidates with an auto_feedback_in column that captures what feedback the generator received before producing this attempt.
  • UI: an accordion per attempt in the Generate panel — show the feedback string at the top, then that attempt's candidates underneath.
  • Bonus: a "what the validator said to the generator" expandable block per attempt.

4. Active validator never ran — candidate_audits has zero rows P1 · env toggle

What's happening: harness/app/actions/generate.ts:172 reads HARNESS_ACTIVE_VALIDATOR === '1' and only runs the agentic auditor when that env is set. It isn't. So the audit machinery (Sonnet + research tools) didn't run on any of the 12 candidates. That's why you only see deterministic regex failures — and why "X unknown" is the tempo signal you get (the regex classifier outputs "unknown" when no main_tempi vocab matches; the LLM auditor would have given a real classification + rationale).

Fix

echo 'HARNESS_ACTIVE_VALIDATOR=1' >> .env
# restart pnpm dev in harness/

This unlocks the second voice on each candidate: a Sonnet audit that reasons about fact validity, voice fit, and tempo classification with research-tool access. That's where the "actually, this paraphrases the bridge mechanic correctly" judgment lives.

5. Tempo classification "unknown" with confidence 0 — dead-end signal P2 · usability

What's happening: the regex classifyTempoBlind only scores against the voice's main_tempi. When the candidate's vocabulary doesn't trigger anchors in any of Infinex's 5 locked tempi, it returns tempo: "unknown" with confidence 0. That's appearing in your UI as "X unknown" — useless feedback.

What it actually means: "the prose didn't sound like any of Commanding · Practical · Sombre · Irradiant · Sociable to the regex." It might be Adream-attitude prose, or Near-attitude prose, or genuine off-placement junk. The regex can't tell you which because it's only scoring against the locked 5.

The LLM auditor (issue 4) would surface this: "I read this as Mobile-Concealed" or whatever. Once the active validator is on, "unknown" becomes a meaningful classification.

UI surfacing

  • When classified_tempo === "unknown", render as "unclassified — prose didn't match any of Infinex's locked tempi" rather than just "X unknown". Includes a small explainer.
  • Show top 2 scoring tempi from the scores object even when no clear winner emerged. Lets you see "almost-Sombre, almost-Commanding, can't decide."
