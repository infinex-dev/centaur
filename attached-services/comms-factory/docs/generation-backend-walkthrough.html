comms-factory — Generation Backend

What actually happens when you click run generator on /cards/[id]. Every claim is anchored to file:line. Generated 2026-06-01 from a code trace of main (working tree).
There are two generation paths and the harness defaults to the newer one. The ACTOR/DIRECTOR path is live by default; the LEGACY two-call path is opt-in via HARNESS_GENERATOR_ARCH=legacy. They share the same regex validator core but have different gating stacks. Read the Actor/Director path first — that's what you're watching in the UI.

TL;DR — the two paths

default — actor

Actor / Director

One LLM (the Actor) does dramaturgical table-work then performs all channels in a single call, emitting ~5 candidates/channel. A second, blind LLM (the Director) forensically classifies each survivor. Retries carry a persistent transcript.

     ├ regex validate() per candidate
     ├ Director → blind audit per survivor

     └ summarizeDirectorNotes() → retry

The stage contract — why the split exists

The whole system is built on one rule from CLAUDE.md: a model that both writes and audits its own output rationalizes slop. So generation and judgment are physically separate processes with separate prompts and separate models.

StageJob — one job onlyLives in
actorProduce. Table-work, then perform. Never self-audits, never names its own tempo.src/actor-director.ts
validatorRegex-grade external gates. No taste, no LLM. First failure blocks.src/validator.ts
directorForensic classification, blind to the Actor's intent. Reads what the prose did, not what the Actor claimed.src/actor-director.ts:285
orchestratorFilter → rank → shape to channel. Picks winners; never generates.src/actor-orchestrator.ts:409

Mirodan grammar underneath: the Actor plays transitive verbs through Working Actions (pressing, punching, flicking…). It is structurally forbidden from declaring a tempo. The Director infers tempo from the finished prose. Tempo is emergent, never assigned — that's the "actor table-work before drafting" discipline encoded as two processes.

What the Actor returns

{
  "warmup":   { ... },              // rehearsal / daily-pages — visible, not final copy
  "table_work": {
     "thesis":        "...",        // one-sentence interpretation of the release
     "through_action":"to ...",     // the transitive intent driving the post
     "obstacle":      "...",
     "lining":        "...",        // the unsaid subtext
     "not_the_point": "...",
     "channel_beat_plans": {        // per channel, a sequence of beats — VERBS only
        "x":   [{ verb, working_action, physical_score, prep_from, micro_objective, ... }],
        "web": [ ... ]
     }
  },
  "performances": {                 // ~5 final-copy options per channel
     "x":   [{ text, rationale, movement_receipt[], deployed_facts_used, not_said }],
     "web": [ ... ]
  },
  "selected_performances": {        // the Actor's own favorite per channel (advisory only)
     "x": { selected_option: 1, selection_rationale: "..." }
  }
}

The load-bearing constraint: the Actor is told to do table-work first, then perform, and to never name, choose, or target a tempo — only to deliberately score Mirodan Working Actions and honor the preparation hierarchy (a Quick action must be prepared by a Sustained one). The movement_receipt maps spans of the final text to the Working Action that produced them — that's the artifact the Director and the validator check against. src/actor-director.ts:340-414

The fact contract — deployed_facts_used & not_said

The per-tweet "do not say" list — what the Actor deliberately withheld, with reasons.

For every single candidate it writes, the Actor attaches two receipts. This is the fact-check anchor: the caption may only assert what's in the card's deployed_facts, and the Actor must account for every fact — either it used it or it consciously declined it.

ReceiptShapeMeaning
deployed_facts_usedstring[]Which card facts this caption actually asserts.
not_said{ fact, reason }[]Card facts the Actor chose not to use in this tweet — each with a one-line reason. This is the per-tweet "do not say."
not_the_pointstringDifferent thing — a table-work-level theme ("what this post is not about"), one per attempt, not per tweet.

The validator's claim contract (src/validator.ts:191-276) enforces it mechanically: every claim must be in deployed_facts_used; every withheld fact in not_said; a fact can't be in both; a card fact must be in one or the other. It's fed in at harness/app/actions/generate.ts:466-469.

Real example from the Hyperliquid-spot card (X, option 1) — the Actor's not_said:

{ "fact": "Hyperliquid spot markets with >$100K 24h volume: 20",
  "reason": "market-count enumeration would project Passion via scarcity; not the through-action" }
{ "fact": "Top markets by 24h volume: HYPE/USDC $130.44M, UBTC/USDC $17.72M, ...",
  "reason": "venue-shilling; reads as hype, not hand-over" }

The Director judge, blind

auditCandidateWithDirector() — src/actor-director.ts:285-301

For each candidate that survives the regex validator, the Director runs as a separate, blind Sonnet 4.6 call (max_tokens=4096). "Blind" = it sees the finished prose and the card, but not the Actor's table-work. Its job is forensic, not aesthetic: classify what the text actually did, then decide if that's legal for the voice.

The four gates it returns

GateAsks
copy_voice_passedDoes the prose fit the voice, and does it cohere under paraphrase?
factual_passedIs every claim supported by the card's deployed_facts?
publication_gate_passedIs this live-product ready (no unshipped state, no placeholder)? Tracked separately from draft quality.
infinex_fit.legalIs the inferred placement + tempo inside the allowed set for this voice?

Draft passes when copy_voice_passed && factual_passed && infinex_fit.legal. DirectorAuditResult — src/actor-director.ts:189-233

What else it reports (diagnostic, not pass/fail)

primary_tempo + confidence, motion_evidence (Weight / Space / Time / Flow poles it read), detected working_actions, movement_receipt_fit (do the Actor's span-claims match what the Director observed?), drive_read (spell / passion / vision …), and notes_for_actor[] — movement-corrective feedback that feeds the retry.

Director discipline (encoded in directorMethodBlock, src/actor-memory.ts:302-368): read the two-factor Variation first; Working Actions support but don't define tempo; drive_read is inferential, not a checklist; don't fail a legal primary-allowed tempo purely on factor shape; a resting Spell without visible Vision is legal; never prescribe "add a second sentence" as a movement rule. These guardrails stop the Director from over-failing single-line UI copy.

The voice spec it reads from

src/voice/infinex.ts is the locked placement (2026-05-12). It declares the inner attitude (Stable), stress (Flow-bound), aspect (Penetrating), drive axis (Spell→Vision), the 5 main tempi (Commanding · Practical · Sombre · Irradiant · Sociable) with their Sustained→Quick motor pairs, the 7 beat-only reserves, plus mirodan_kernel and drive_table. src/voice/types.ts defines the CharacterSpec / Tempo / WorkingAction shapes; laban.ts holds the prep-hierarchy machinery; cream.ts / nigel.ts are sibling voices (same shape, different placement).

The same voice spec also feeds the legacy two-call generator's system prompt via buildSystemPrompt() → buildV2SystemPrompt() src/generator.ts:340-625. One voice spec, two prompt assemblers — that's the portability the project cares about.

Selection & ranking

pickRankedPassingByChannel() :409 · scoreCandidateForSelection() :458 — src/actor-orchestrator.ts

Per channel: keep only candidates that passed regex and the Director draft gate, then score and rank.

base   = script(100) + factual(100) + copy_voice(100) + legal(100)        // max 400
bonus  = publication_gate(+8) + actor_recommended(+6)
       + distinct_image(+4) + channel_fit(+4)
penalty= image_cluster_repetition(-10) + repetition(-N) + rail_tokens(-N)
       + voice_issues*12 + factual_issues*20 + operator_preferences

"Image clusters" (:583-593) — wall-boundary, rail-plumbing, account-surface, market-object, spark-object, fact-literal — get penalized for repeating across candidates, so a batch doesn't converge on the same metaphor. Highest score wins the channel.

Known TODO: the legacy orchestrate() path still uses naive "first-pass-wins" ranking src/orchestrator.ts:290. The actor path's scoring above is the real ranker.

The retry loop

If a channel has no passing pick, the orchestrator synthesizes Director notes and runs the Actor again — up to 3 attempts — as a persistent conversation (the Actor sees its prior output).

summarizeDirectorNotes(attempt, channels, records, picks)   // src/actor-orchestrator.ts
  └ buildDirectorNotesMessage()                              // actor-director.ts:478
       ## Preserve unless explicitly contradicted
       - through_action / beat_plan / working_actions / objective_verbs
       ## Change lanes
       - copy / facts / voice / movement

The principle (:479-484): preserve the table-work and motor unless the Director explicitly failed it; only revise the copy, facts, and local line issues. Stops the Actor from re-deriving the whole interpretation each round. After 3 exhausted attempts it returns { exhausted:true, picks: bestSoFar }.

The gate sequence — what blocks, in what order

Two stacks, depending on path. Both start with the same regex core.

actor / director stack
  1. Card approved? API key present?
  2. Regex validator — validate()
  3. Actor movement checks (tempo not declared, receipt present, prep hierarchy)
  4. Director audit — blind LLM, 4 gates
  5. Score + rank → pick
legacy stack
  1. Card approved? API key present?
  2. Regex validator — validate()
  3. Active LLM validator — with research tools
  4. Length gate (X 280 / web 140 / in-product 80)
  5. History guards — freshness vs shipped copy
  6. First survivor per channel → pick

Both paths feed audits into candidate_audits / director_audits for operator inspection. The regex layer is blocking and first in both — first failure stops evaluation, so the expensive LLM never runs on obviously-bad copy.

Claim contract — the fact-check anchor :191-348

The caption cannot assert anything outside the card's deployed_facts. The validator enforces receipts: every claim must appear in deployed_facts_used; everything deliberately withheld in not_said; a fact can't be in both; a card fact must be in one or the other; and auditUnsupportedClaims() (:305) extracts URLs/numbers/claims and fuzzy-matches them back to the card.

Beat-sequence audit :441-651

Splits copy into beats, runs a deterministic heuristic tempo classifier (classifyTempoBlind :513 — scores each paragraph against the voice's vocab anchors + motor vocabulary; short copy caps confidence at 0.6), and checks the Laban prep→release motor hierarchy (auditPrepHierarchy :460). A beat passes if it classifies as 'unknown' or into an allowed tempo. Note: this heuristic classifier is separate from the Director's LLM classification — it's the cheap regex-grade pre-read.

Active LLM validator legacy stack

src/validator-active.ts:85-208 · default model claude-sonnet-4-6

On the legacy path, after regex passes, an agentic validator runs (up to 6 tool turns). It can call research/fact-check tools to verify claims against platform code / registries before voting, then emits a verdict via an audit_pass / audit_fail tool call:

{ passed, deterministic, llm:{ feedback, voice_issues[], factual_issues[],
   independent_classification:{ tempo, motifs[], detected_drive, confidence } },
   research_trace[], active_turns, truncated, reason }

Behavior locked by recent commits: LLM validator is the default gating voice, zero-tolerance em-dash, and it fails loud when the LLM is unavailable rather than silently falling back to regex. Disable only with HARNESS_ACTIVE_VALIDATOR=0 (not recommended). If it hits max turns without a verdict → fail with active-validator-max-tool-turns. :195-207

The actor/director path doesn't run this validator — the Director is its judgment layer. The active validator is the legacy path's equivalent. Conceptually parallel; different implementations.

How this fits a Slack bot or a new harness

The harness is explicitly a temporary review tool, not the product — production surfaces are the Slack bot + an MCP server. The good news from this trace: the entire pipeline is harness-independent. Everything load-bearing lives in src/; the harness is a thin Next.js shell over it.

Reusable core — call from anywhere

  • orchestrateActorDirectorWithRetries() — the whole default pipeline, one call. Takes a ReleaseCard + channels, returns picks + full audit trail.
  • validate() — the regex gate, pure + synchronous. Cheapest reuse; ideal for an MCP /validate tool or a Slack ad-hoc check.
  • generate() — the legacy two-call generator if you want the cheaper non-director path.
  • The CLI (src/cli.ts) already wires generate / validate subcommands — it's the reference for how a non-harness caller drives the stages.

Harness-specific — don't port

  • SQLite persistence (actor_attempts, candidates, director_audits, pipeline_runs) — a Slack bot wants its own store or none.
  • The onEvent checkpoint streaming → React panels. A Slack bot would map these to thread messages instead.
  • Operator decision buttons (approve/edit/retry/reject) — the harness's whole reason to exist (agreement metrics). The Slack/MCP fast lane skips this; the ship gate stays human regardless.

Concrete shapes

Slack bot (fast lane): slash command → build a ReleaseCard from the event → orchestrateActorDirectorWithRetries(card, channels, {maxAttempts: 1-2}) → post the top pick per channel into a thread with the Director verdict as context → 👍 reaction is the ship gate. The retry loop and Director notes become thread replies if the operator asks for another pass.

MCP server: three tools mapping straight onto existing functions — /generate → orchestrateActorDirectorWithRetries, /validate → validate (+ optionally the active validator), /audit → auditCandidateWithDirector on arbitrary text. The Director-on-arbitrary-text call is the highest-value standalone tool: it audits hand-written copy, not just generated.

Decision you'll face: which gating stack does the fast lane use? The actor/director stack is richer (blind Director, scoring) but is 1 Actor call + N Director calls per attempt. The legacy stack's active-validator+history-guards is lighter and already has freshness baked in. For a high-frequency Slack lane you may want regex + active-validator + history-guards (cheap, freshness-aware) and reserve the full Actor/Director run for the deep-review harness lane. That's a real fork, not a config flag — worth deciding before wiring either surface.
