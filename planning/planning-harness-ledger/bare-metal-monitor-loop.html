bare_metal monitor loop

A local-first plan for a Centaur-native durable workflow that watches a bare_metal run, detects failure or progress, and brings an agent into Slack only when judgement or operator action is needed.

Starting constraint

Keep the first version on one machine. No coder-box registry, SSH, or network policy work until the loop and Slack operating model are proven.

What we are building

The first deliverable is a Centaur workflow named bare_metal_monitor. It runs as a durable loop: poll a bare_metal run, checkpoint the observation, classify the state, sleep, and resume later. The workflow is native to Centaur; the endpoint it polls belongs to bare_metal.

The monitor is deterministic by default. It should not ask an LLM to inspect every poll. It calls an agent only when there is a meaningful transition: failure, completion, stale progress, budget threshold, or operator-requested triage.

Lowest-risk slice: attach to an existing local bare_metal run and detect a failing task. Do not start PM2, allocate ports, submit plans, or talk to a remote coder box in the first pass.

Architecture split

Layer Owns First implementation
Centaur workflow Durable loop, checkpoints, wake schedule, escalation policy. workflows/bare_metal_monitor.py
Centaur tool HTTP calls to bare_metal and response normalization. tools/infra/bare_metal_http
bare_metal server Run state, task state, cost, errors, pause/resume/abort. Existing server on http://localhost:4016 or another local port.
Slack thread Operator surface: start monitor, see status, approve next action. Existing Slackbot path through slack_thread_turn.

Centaur-native loop

for poll_index in range(inp.max_polls):
    snapshot = await ctx.call_tool(
        "bare_metal_http",
        "get_run",
        {"base_url": inp.base_url, "run_id": inp.run_id},
    )

    decision = classify(snapshot, previous_snapshot)
    if decision.requires_agent:
        await ctx.run_agent("triage", text=decision.prompt, delivery=inp.delivery)

    if decision.terminal:
        return decision.summary

    await ctx.sleep("poll_delay", timedelta(seconds=inp.poll_interval_seconds))

Each ctx.call_tool and ctx.sleep is checkpointed. On replay, Centaur skips completed checkpoints and continues from the next unfinished poll.

Monitor input

The monitor should receive enough information to attach to a run and know when to speak up. The first version does not need to know how the run was created. 

{
  "base_url": "http://host.docker.internal:4016",
  "run_id": "run_abc123",
  "poll_interval_seconds": 30,
  "stale_after_seconds": 300,
  "max_polls": 200,
  "notify_on": [
    "completed",
    "failed",
    "stale",
    "task_transition"
  ],
  "delivery": {
    "platform": "slack",
    "channel": "C123",
    "thread_ts": "1773364194.179929"
  }
}

Poll source

GET {base_url}/api/v1/runs/{run_id} is a bare_metal API call. The Centaur tool wraps it and returns a compact normalized snapshot.

Time gate

poll_interval_seconds controls the workflow sleep. This is not an in-memory daemon sleep; it is a persisted Centaur workflow wake time.

Snapshot fields

  • status: running, paused, completed, failed, or aborted.
  • failureReason and error, when present.
  • totals: cost, completed tasks, failed tasks, total tasks, duration.
  • tasks: task id, title, status, cost, error.
  • A derived fingerprint used to detect progress between polls.

MVP scope

Add a minimal bare_metal HTTP tool

Create a Centaur tool with get_run(base_url, run_id). For the local demo, allow only loopback-style URLs and return a normalized subset of the run response.

Add the durable monitor workflow

Create bare_metal_monitor as an external workflow. It polls, classifies, sleeps, and exits on terminal states.

Run a local failing demo

Start a small bare_metal run with three tasks where one task is designed to fail. The monitor should detect the failed run and produce a final summary.

Connect Slack

Start the workflow from a Slack thread or let an agent start it after being given a run URL. The workflow posts back into the same thread when the run changes state.

Setup steps

1. Start Centaur locally

cd /Users/kain/Sites/Centaur
just up

2. Start or reuse a local bare_metal server

For the current planning-harness run, ports 4016 and 4017 are already allocated in the hard port map. 

pm2 logs bare-metal-centaur-planning-harness-ledger-diverge --lines 20 --nostream
curl -sS http://localhost:4016/api/v1/runs | jq

3. Verify Centaur API can reach the bare_metal API

From inside the API deployment, localhost is the API pod, not the laptop host. Validate the host route before wiring the workflow. 

kubectl exec -n centaur deploy/centaur-centaur-api -- \
  curl -sS http://host.docker.internal:4016/api/v1/runs | jq

4. Register the tool and workflow

Put the tool under tools/infra/bare_metal_http for the base prototype, or in an overlay once the contract stabilizes. Put the workflow under workflows/bare_metal_monitor.py. 

kubectl exec -n centaur deploy/centaur-centaur-api -- \
  curl -fsS http://localhost:8000/health/tools | jq

5. Start the monitor workflow

kubectl exec -n centaur deploy/centaur-centaur-api -- curl -sS \
  -X POST http://localhost:8000/workflows/runs \
  -H "Content-Type: application/json" \
  -d '{
    "workflow_name": "bare_metal_monitor",
    "eager_start": true,
    "input": {
      "base_url": "http://host.docker.internal:4016",
      "run_id": "RUN_ID",
      "poll_interval_seconds": 30,
      "stale_after_seconds": 300,
      "max_polls": 200
    }
  }' | jq

6. Inspect workflow state

kubectl exec -n centaur deploy/centaur-centaur-api -- curl -sS \
  http://localhost:8000/workflows/runs/WORKFLOW_RUN_ID | jq

Getting the agent in Slack

Slack should be the operator surface, not the execution engine. The agent can help start or inspect the monitor, but the durable workflow owns the loop.

Manual workflow start

An operator starts the workflow with a curl/API call and includes Slack delivery metadata. The monitor posts summaries into the chosen channel and thread.

Agent-assisted start

In Slack, mention Centaur and provide the bare_metal run URL or run id. The agent extracts the run id, confirms the base URL, and starts bare_metal_monitor.

Suggested Slack interaction

@Centaur monitor bare_metal run
base_url: http://host.docker.internal:4016
run_id: RUN_ID
poll: 30s
alert here when it fails or completes

Agent prompt contract

  • Confirm the run id and base URL before starting a monitor.
  • Do not poll from the agent conversation. Start a workflow.
  • Summarize the workflow id and how to inspect it.
  • When the monitor escalates, explain the failure and propose one next operator action.

Failing demo run

The demo should prove the loop, not the full operator workflow. Use a tiny plan with three tasks where task two fails deterministically. The monitor should observe the transition from running to failed and report the failed task and error.

Task Expected behavior Monitor assertion
DEMO-001 Creates a small file and succeeds. Task moves to done.
DEMO-002 Runs a failing gate or exits non-zero. Run becomes failed with tasksFailed > 0.
DEMO-003 Depends on the failed task or remains pending. Monitor summary identifies incomplete downstream work.

Acceptance criteria

  • Centaur workflow run checkpoints at least two polls.
  • The workflow sleeps between polls instead of hot-looping.
  • The workflow exits when bare_metal reports a terminal status.
  • The final output names the failed task, failure reason, and run URL.
  • Slack delivery works when delivery.platform is slack.

After the local demo

Once the local loop works, add control-plane features in small increments. Remote coder-box support should wait until the local behavior is boring.

Add richer notifications

Notify on task transitions, stale progress, cost thresholds, failure, completion, and pause events.

Add pause/resume/abort methods

Extend the tool to call bare_metal control endpoints only after explicit operator approval.

Add start/submit flow

Fold in the checked-in bare-metal/execute workflow: preflight, PM2 backend, UI, config review, submit.

Add remote machine support

Only after local monitoring works, introduce a coder-box transport with an allowlist for hosts, repo roots, ports, and commands.
