# E2E Testing Matrix

This is the map for agents adding or reviewing Centaur E2E coverage. Keep it
focused on product behavior: what a client or user can observe, and the service
paths that must work for that behavior to be reliable.

Use the smallest test that proves the contract. Add full-stack coverage when
the API worker, workflow worker, sandbox, harness, or tool runtime is part of
the risk. Do not add new coverage against legacy `/agent/connect`; current
paths use durable workflow or agent control-plane calls and replay output via
`/agent/threads/{thread_key}/events`.

When adding a test, update this matrix so the next agent can see where coverage
lives.

## Command

- `pnpm run e2e:test`: run the E2E suite.

| Contract | What to test | Existing coverage | Gap | Acceptance check |
| --- | --- | --- | --- | --- |
| Client turn returns one visible answer | Send a user turn through a client path and verify the terminal text reaches the user once. | `services/slackbot/test/runtime-control.test.ts`; `services/slackbot/test/integration.test.ts` has skipped integration coverage. | Add a direct SDK/API full-stack turn that uses `spawn -> message -> execute -> events` without Slack. | One reply is posted or returned, final text contains the requested result, and no generic pre-start failure appears. |
| Follow-up turn does not create duplicate work | Send a new user message while a prior execution is active. | `services/slackbot/src/index.test.ts`; `services/slackbot/src/slack/dedup.test.ts`; `services/api/tests/test_integration.py`; `services/api/tests/test_workflows.py`. | Add full-stack active-execution coverage if the real harness/API steering interaction changes. | The follow-up either steers the active execution or starts one replacement workflow; it does not start duplicate work for the same message. |
| Stop requests are clean | Stop a long generation and verify stale output is not delivered. | `services/slackbot/test/runtime-control.test.ts`; `e2e/scenarios/slackbot-history-stop-race.e2e.test.ts` is an expected-failing regression target. | Fix and unmark the stop/history race when that behavior is ready. | Prior execution is cancelled, the follow-up response is friendly, and stale nonce-heavy output is not delivered. |
| Replaced or cancelled executions do not post stale final delivery | Claim final-delivery rows after replacement or cancellation. | `services/api/tests/test_agent_control_plane.py`. | Keep this current when steering or final-delivery semantics change. | Suppressed delivery is marked delivered without a post; the latest visible cancellation posts one friendly message. |
| Streaming and final delivery do not double-post | Simulate an outbox row for an execution that is still streaming locally. | `services/slackbot/src/centaur/final-delivery.test.ts`; `services/api/tests/test_agent_control_plane.py`. | Maintain normal-path assertions around ack ordering. | While streaming, outbox delivery is not posted; after streamed delivery, final delivery is acked once before cosmetic edit/title work. |
| Durable replay uses stored terminal output | End the event stream with no unseen terminal event and hydrate from `GET /agent/executions/{id}`. | `services/slackbot/test/runtime-control.test.ts`. | Add the same reconnect assertion through the API client once direct SDK/API E2E exists. | The client receives stored result text, not "Agent request failed before execution started." |
| Delivery failures degrade to readable output | Force expected delivery failures such as restricted destinations, processing failures, `msg_too_long`, or `invalid_blocks`. | `services/slackbot/test/bot.test.ts`; `services/slackbot/test/runtime-control.test.ts`; `services/slackbot/test/slack-stream.test.ts`. | Add cases when a new delivery target or error path changes behavior. | Plain fallback posts preserve the final answer/table text; failed fallback posts are not acked. |
| Markdown and tables stay valid | Render representative final answers with headings, code fences, links, tables, and long text. | `services/slackbot/test/slack-markdown.test.ts`; `services/slackbot/test/runtime-control.test.ts`. | Add cases when introducing new rendered block shapes. | Code languages survive, tables render or flatten safely, and content stays inside client limits. |
| Dashboard blocks do not leak raw syntax | Convert KPI, table, and chart dashboard blocks before delivery. | `services/slackbot/test/dashboard-to-slack.test.ts`; `services/slackbot/test/runtime-control.test.ts`. | Add cases for each new dashboard block type before using it in agent output. | Supported dashboard blocks become readable markdown or files; raw fenced dashboard syntax appears only when intentionally unparseable. |
| File links are usable | Rewrite sandbox-local `file://` links before delivery. | `services/slackbot/test/runtime-control.test.ts`. | Keep assertions current when harness event metadata changes. | Delivered text contains stable repo URLs and does not expose sandbox-local paths. |
| Attachments and history reach workflows | Convert files into content blocks and pass prior thread messages into workflow execution. | `services/slackbot/test/bot.test.ts`; `services/slackbot/test/attachment-refetch.test.ts`; `services/slackbot/test/runtime-control.test.ts`; `services/api/tests/test_attachments.py`. | Add full-stack attachment coverage for the direct API path and any non-Slack client path. | Current message is not duplicated into history, older user context is included, and image/document attachments become content blocks or attachment refs. |
| Runtime errors are friendly and sanitized | Render failed terminal states and provider errors. | `services/slackbot/test/runtime-control.test.ts`. | Keep this close to terminal-result rendering unless real harness classification changes. | User sees friendly retry/cancellation copy; secrets and internal paths are redacted; successful answers that mention errors do not trigger alerts. |
| Workflow API runs and resumes durable work | Start, inspect, cancel, and wake workflow runs. | `services/api/tests/test_workflows.py`; `services/api/api/workflows/agent_turn.py`; `services/api/api/workflows/slack_thread_turn.py`. | Add full-stack workflow E2E for a non-Slack `agent_turn` and a workflow event wakeup. | Runs expose status/checkpoints, resume from durable state, and terminal output is observable through the client. |
| Tool calls are discoverable and callable | Discover tools and call representative read-only methods through the API. | `services/api/tests/test_tool_manager.py`; tool-specific tests under `tools/**/tests`. | Add an E2E tool smoke set for registered read-only tools with safe inputs. | Tool discovery returns expected methods; safe tool calls return data or classified upstream/auth errors without breaking the agent turn. |
| Sandbox and harness protocol remains compatible | Exercise content blocks, attachment refs, harness output normalization, and Amp wrapper behavior. | `services/api/tests/test_harness_protocol.py`; `services/api/tests/test_amp_wrapper.py`; `services/api/tests/test_stream_exec.py`. | Add coverage before enabling new harnesses or changing content-block translation. | The sandbox receives valid harness input, emits normalized events, and terminal results are projected consistently. |
| Auth and scope boundaries hold | Use scoped API keys across agent, workflow, and tool surfaces. | `services/api/tests/test_check_scope.py`; `services/api/tests/test_sandbox_tokens.py`; `services/api/tests/test_agent_control_plane.py`. | Add E2E checks for external client keys once direct API E2E is added. | Allowed scopes succeed, disallowed scopes fail clearly, and sandbox tokens cannot exceed their scope. |
